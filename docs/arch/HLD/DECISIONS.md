## ADR-001: Privacy-First — Agent 전용 암호화 콘텐츠

**날짜**: 2026-03-01
**상태**: 확정

### 결정

공유된 프로젝트 콘텐츠는 RSA-4096으로 암호화하여 AI Agent만 접근 가능하게 한다. 인간 사용자(원본 소유자 포함)도 다른 사람의 공유 콘텐츠 원문을 읽을 수 없다.

### 근거

- 사용자가 자신의 개인적 여정, 실수, 민감한 결정 내용을 공유하는 데 심리적 장벽을 낮춤
- 콘텐츠를 "인간이 읽는 것"이 아니라 "AI가 학습하는 것"으로 포지셔닝 → 공유 의향 증가
- Agent가 지식을 합성하여 일반적 통찰만 사용자에게 전달하므로 실질적 프라이버시 보장

### 트레이드오프

- (+) 사용자 신뢰도 및 공유 의향 증가
- (-) Agent 전용 접근 구현 복잡도 증가
- (-) 콘텐츠 품질 검수가 사람이 아닌 모델 기반으로만 가능

## ADR-002: Local-First 저장소 (Markdown + PageIndex 서버)

**날짜**: 2026-03-01
**상태**: 확정

### 결정

모든 사용자 원본 데이터는 로컬 디바이스에 Markdown 파일로 저장한다. 섹션 단위 검색 인덱스는 로컬 PageIndex 서버(Python/Docker)가 담당한다. Hub에는 익명화된 지식 그래프만 업로드한다.

### 인덱싱 아키텍처 변천

| 버전 | 방식 | 변경 이유 |
|------|------|----------|
| v1 | better-sqlite3 + sqlite-vec | 초기 설계 |
| v2 | @lancedb/lancedb | sqlite-vec native addon → OpenClaw Jiti 로더 불가 |
| v3 | PageIndex 로컬 서버 (HTTP) | LanceDB도 native addon → Jiti 불가 + VectifyAI PageIndex가 Python 전용 라이브러리 → 사이드카 구조가 최적 |

### 근거

- 인터넷 연결 없이도 전체 로컬 기능 사용 가능
- Markdown은 사용자가 직접 편집 가능하고 OpenClaw 기존 구조와 호환
- VectifyAI PageIndex: Vectorless LLM 추론 기반 검색 — 임베딩 없이 헤더 트리 구조만으로 높은 검색 품질 제공
- OpenClaw 플러그인은 Jiti 로더로 실행 → native addon 불가 → Python 사이드카가 유일한 현실적 선택

### 트레이드오프

- (+) 오프라인 동작, 사용자 데이터 주권, OpenClaw 통합 용이
- (+) 임베딩 비용 없음 (LLM 추론으로 대체, PageIndex 서버 내부 처리)
- (-) PageIndex 서버 미실행 시 검색 불가 (graceful degradation: 빈 결과 반환, 파일 저장은 정상)
- (-) 디바이스 간 동기화 미지원 (Phase 2 고려)

## ADR-003: 평판 기반 접근 제어 (금전 거래 없음)

**날짜**: 2026-03-01
**상태**: 확정

### 결정

프리미엄 콘텐츠 접근은 평판 점수(기여 기반)로만 제어한다. 크레딧, 토큰, 구독료 시스템을 도입하지 않는다. 단, 일회성 접근 구매는 보조 옵션으로 유지.

### 근거

- 지식 공유 커뮤니티의 선순환: 기여 → 평판 → 더 나은 도구 → 더 좋은 기여
- 금전 장벽 없이 누구나 참여 시작 가능
- 토큰/크레딧 시스템의 투기/악용 위험 제거

### 트레이드오프

- (+) 선순환 커뮤니티 생태계, 진입 장벽 없음
- (-) 초기 신규 사용자의 가치 획득 속도 느림
- (-) 빠른 수익화 어려움 (장기 관점 필요)

## ADR-004: 차등 프라이버시 (ε = 0.1 ~ 1.0)

**날짜**: 2026-03-01
**상태**: 확정

### 결정

익명화 후에도 수치 데이터(예산, 기간)에 보정된 노이즈를 추가한다(Differential Privacy, ε = 0.1 ~ 1.0). 집계 통찰은 n ≥ 10 소스에서만 제공한다.

### 근거

- 단순 NER 제거만으로는 재식별 공격 가능 (예: 특정 예산 + 특정 날짜 조합)
- ε ≤ 0.1: 높은 프라이버시, 낮은 유용성 / ε = 1.0: 낮은 프라이버시, 높은 유용성 → 0.5 기본값
- n < 10 제한으로 소규모 그룹 식별 위험 차단

### 트레이드오프

- (+) 강력한 수학적 프라이버시 보장
- (-) 특히 낮은 ε에서 데이터 유용성 저하
- (-) 구현 복잡도 증가

## ADR-005: 연합 학습 (Federated Learning)

**날짜**: 2026-03-01
**상태**: 확정

### 결정

집계 패턴 수집에 Federated Learning을 사용한다. 원본 데이터는 디바이스를 절대 떠나지 않으며, 암호화된 그래디언트만 Hub로 전송한다. 집계 방식은 SMPC 또는 동형암호 사용.

### 근거

- "이 경험이 공유됩니다" 대신 "이 패턴이 학습됩니다"로 사용자 수용성 향상
- 원본 데이터 중앙 집중 없이 커뮤니티 집합 지식 생성 가능

### 트레이드오프

- (+) 최고 수준의 프라이버시 보장, 중앙화 위험 없음
- (-) 구현 복잡도 매우 높음 (PySyft, TensorFlow Federated)
- (-) 모든 환경에서 클라이언트 부하 발생 가능

## ADR-006: PARA 방법론 채택

**날짜**: 2026-03-01
**상태**: 확정

### 결정

Projects / Areas / Resources / Archive 4개 카테고리로 메모리를 조직화한다.

| 카테고리 | 정의 |
|----------|------|
| Projects | 마감일이 있는 단기 목표 |
| Areas | 지속적으로 관리해야 하는 책임 영역 |
| Resources | 미래에 유용할 수 있는 참고 자료 |
| Archive | 완료되거나 비활성화된 항목 |

### 근거

- Tiago Forte의 PARA는 널리 검증된 지식 조직화 방법론
- 명확한 카테고리 경계로 Intent Classifier 구현 단순화
- 사용자가 이미 PARA 개념에 익숙한 경우 진입 장벽 최소화

### 트레이드오프

- (+) 검증된 방법론, 직관적 조직화
- (-) PARA 방법론에 익숙하지 않은 사용자는 초기 학습 필요
- (-) 일부 콘텐츠가 카테고리 경계에 걸릴 수 있음

## ADR-007: `para-memory`는 `memory-core`와 공존 (슬롯 비경쟁)

**날짜**: 2026-03-01
**상태**: 확정

### 결정

`para-memory` 플러그인은 `openclaw.plugin.json`에 `"kind": "memory"`를 선언하지 않는다.
OpenClaw의 메모리 슬롯(`plugins.slots.memory`)을 점령하지 않고 `memory-core`와 공존한다.

### 근거

- `memory-core`는 대화 컨텍스트 메모리(MEMORY.md) 관리 — PARA 플러그인이 대체할 이유 없음
- `para-memory`의 역할은 "메모리 저장 방식"이 아니라 "PARA 방법론 기반 조직화 레이어" 제공
- 두 플러그인이 독립적인 도구를 등록하므로 Agent가 상황에 맞게 선택 가능

### 도구 분리

| 도구 출처 | 도구 | 역할 |
|----------|------|------|
| `memory-core` | `memory_search`, `memory_get` | 일반 대화 메모리 검색/조회 |
| `para-memory` | `classify_intent`, `search_project`, `check_transition` | PARA 프로젝트 구조 검색/관리 |

### 트레이드오프

- (+) 책임 분리 명확, `memory_search`/`memory_get` 미구현으로 구현 단순화
- (+) `memory-core` 비활성화 없이 PARA 기능 추가 가능
- (-) Agent가 두 메모리 시스템을 구분해서 호출해야 함 (시스템 프롬프트 가이드 필요)

## ADR-008: Hub 1차 배포는 Docker Compose (Kubernetes 이후 단계)

**날짜**: 2026-03-01
**상태**: 확정

### 결정

Hub의 초기 배포는 Docker Compose + 단일 서버로 구축한다. Kubernetes(EKS) 전환은 실제 트래픽 검증 후 Phase 2+에서 진행한다.

### 근거

- Phase 1 목표는 기능 검증이지 고가용성이 아님 → 단순한 배포 구조가 개발 속도를 높임
- docker-compose.yml 서비스 정의가 이후 Kubernetes Helm 차트의 기반이 됨 (마이그레이션 비용 최소화)
- 초기 사용자 수가 Kubernetes 운영 복잡도를 정당화할 수준이 아님
- MinIO: S3 호환 API 제공 → AWS S3로 교체 시 애플리케이션 코드 변경 불필요

### 단계별 배포 전략

| 단계 | 배포 방식 | 전환 조건 |
|------|-----------|-----------|
| Phase 1 | Docker Compose (단일 서버) | - |
| Phase 2 | Kubernetes EKS | DAU 1,000+ 또는 응답시간 SLA 위반 |

### 트레이드오프

- (+) 초기 개발/배포 복잡도 최소화, 빠른 이터레이션
- (+) docker-compose → Kubernetes 마이그레이션 경로 명확
- (-) 단일 서버 → 단일 장애점 (Phase 1에서 SLA 99.9% 미보장)
- (-) 수평 확장 불가 (Phase 1 기간 동안)
